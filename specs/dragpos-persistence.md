# DragPos Persistence & Conflict Resolution Spec

This document outlines current limitations and future improvements for how `dragPos` (image/element positions) are persisted.

## Current Behavior

- `dragPos` stored in each slide's frontmatter within `slides.md`
- Interactive edits trigger 500ms debounced save via HTTP POST
- Server patches frontmatter and rewrites entire markdown file
- Single-writer assumption: conflicts occur if editing `slides.md` externally

## Known Issues

### File-level conflicts
- Dragging an image rewrites `slides.md`
- External editor changes to `slides.md` can race with drag saves
- No conflict detection or merge logic
- Workaround: avoid concurrent edits, or use undo

### Multi-session conflicts
- Multiple browser tabs/windows share the same file
- No coordination between sessions
- Last-write-wins semantics

## Future: Separate DragPos File

Store positions in a sidecar file instead of inline frontmatter.

### Proposed Format

```yaml
# .slidev/dragPos.yaml
# Auto-generated by Slidev - edit positions interactively

1:  # slide 1
  img-mario: 100,50,200,150
  img-luigi: 300,50,200,150,45  # with rotation

2:  # slide 2
  img-peach: 50,100,200,150
```

### Config

```yaml
# slides.md frontmatter
---
dragPosFile: .slidev/dragPos.yaml  # default
# or: dragPosFile: false  # inline in frontmatter (current behavior)
---
```

### Behavior

1. **Load**: Read sidecar file, merge with any inline frontmatter (inline takes precedence)
2. **Save**: Write to sidecar file only, never patch slide frontmatter for dragPos
3. **Build**: Optionally inline all positions back to frontmatter for distribution

### Pros
- Separates content edits from position edits
- Can be `.gitignore`d if positions are ephemeral
- Single file instead of scattered frontmatters
- Survives slide content refactoring

### Cons
- Slide reordering changes keys (could use stable slide IDs instead)
- Less self-contained (need both files)
- Migration path for existing decks

## Future: Multi-Writer Support (Google Slides style)

Real-time collaborative editing with conflict-free semantics.

### Architecture Options

#### Option A: Operational Transform (OT)
- Each edit is an operation (move, resize, rotate, crop)
- Server transforms concurrent operations
- Complex but battle-tested (Google Docs/Slides)

#### Option B: CRDT-based
- Conflict-free replicated data types
- Each field is a last-writer-wins register with vector clocks
- Simpler than OT, good for position data
- Libraries: Yjs, Automerge

#### Option C: Lock-based
- Acquire lock on element before editing
- Show "User X is editing" indicators
- Simpler but less fluid UX

### Required Infrastructure

1. **WebSocket server** for real-time sync
2. **Authentication** to identify users/sessions
3. **Presence** to show who's viewing/editing
4. **Persistence** layer (file, database, or both)
5. **Cursor/selection sync** for awareness

### MVP Scope

1. Single-file lock (only one editor at a time)
2. Session awareness (show active editors)
3. Warn before overwriting concurrent changes

### Full Scope

1. Per-element granular locking or CRDT
2. Real-time cursor/selection sync
3. Undo/redo across sessions
4. Offline support with sync on reconnect
5. Version history / time travel
